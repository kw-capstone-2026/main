# 서울시 상가 폐업 예측 파이프라인 기술 명세 및 실험 가이드

이 문서는 서울시 상권 데이터를 활용하여 개별 점포의 폐업 여부를 예측하는 머신러닝 파이프라인의 전체 구조와 실험 가이드를 제공합니다. 팀원들은 이 문서를 바탕으로 피처 엔지니어링과 모델 튜닝을 진행하여 예측 성능을 극대화할 수 있습니다.

---

## 1. 데이터 소스 및 활용 목적 (Data Sources)
모델은 총 5종의 이기종 데이터를 수집하여 융합합니다.

1. **DS1 (상가 정보 - 공공데이터포털)**: 현재 영업 중인 점포의 상세 데이터. (클래스 0: 영업 중 라벨링에 주로 사용)
2. **DS2 (서울시 상권분석 - 매출/유동인구)**: 상권의 거시적 경제 지표. 개별 점포가 속한 지역의 구매력과 유동성을 평가하는 데 사용.
3. **DS3 (서울시 인허가 데이터)**: 과거 폐업 및 오픈 이력 전체. (클래스 1: 폐업 라벨링의 핵심 소스)
4. **DS4 (기초구역도 SHP)**: 서울시를 5,665개의 미세 블록으로 쪼갠 폴리곤(다각형) 지도 데이터. 공간 조인(Spatial Join)의 기준점.
5. **DS5 (격자 생활인구)**: 250m 단위의 초정밀 인구 밀집도. 상권의 주야간 활성도 유추에 활용.

---

## 2. 데이터 통합 및 파이프라인 관리 (Data Integration & Management)
전체 파이프라인은 모듈화되어 있으며 `main.py`가 오케스트레이터 역할을 합니다.

- **`src/data_acquisition.py`**: API 통신 전담. 
  - *작업 내용*: DS3에서 과거 폐업 데이터를 역순으로 2만 건 수집하고, DS1에서 최신 영업 데이터를 보강하여 **클래스 불균형을 1차적으로 완화**한 하이브리드 데이터셋을 구축합니다.
- **`src/spatial_processor.py`**: 지리 정보 처리.
  - *작업 내용*: 위경도(WGS84)와 중부원점(TM)이 혼재된 원본 좌표를 일괄 EPSG:5179로 변환하고, 점포의 Point 좌표를 기초구역도 Polygon 안에 집어넣는 Spatial Join을 수행합니다.
- **`src/feature_merger.py`**: 파생 변수 생성 및 데이터 조인.
  - *작업 내용*: 점포 데이터에 기초구역(`BAS_ID`)을 매개로 매출, 인구 데이터를 결합합니다.
- **`src/models/baseline_xgboost.py`**: 모델 학습 및 평가.

---

## 3. 핵심 전처리 과정 (Preprocessing)
1. **결측치 및 매핑 실패 데이터 제거**: 서울 외곽이나 좌표 오류로 기초구역(BAS_ID) 매핑에 실패한 점포는 과감히 분석 대상에서 제거(`dropna`)하여 공간 데이터의 무결성을 유지합니다.
2. **날짜 데이터 파싱 (Temporal Parsing)**: `인허가일자`(예: YYYY-MM-DD) 문자열을 `datetime` 객체로 파싱하여 점포의 오픈 연도(`open_year`)를 추출합니다.
3. **결측치 제로화**: 매출이나 유동인구가 잡히지 않는 신생/소형 상권의 결측치는 `fillna(0)` 처리하여 수치형 변환 에러를 방지했습니다.

---

## 4. 모델링 패러다임: 왜 Store-level Classification 인가?
초기에는 **"특정 블록(BAS_ID)의 평균 폐업률 예측(Regression)"**을 시도했습니다. 하지만 점포가 1~2개뿐인 블록의 경우 폐업률이 0.0 아니면 1.0으로 극단화되어 모델이 쉽게 과적합(Overfitting)되는 문제가 발생했습니다.

이를 극복하기 위해 **"이 개별 점포가 폐업할 것인가?(Classification)"**로 문제를 재정의했습니다.
- **타겟 변수**: `is_closed` (0: 영업 중, 1: 폐업)
- **분류기**: XGBoost Classifier

---

## 5. 피처 엔지니어링 기법 (Feature Engineering)
단순한 수치 결합을 넘어, 상권의 본질을 꿰뚫는 파생 변수를 생성했습니다. 팀원들은 이 부분을 집중적으로 고도화해야 합니다.

- **`store_age` (점포 연차)**: `현재 연도 - open_year`. 폐업에 가장 결정적인 영향을 미치는 변수.
- **`local_competitors`**: 동일한 BAS_ID 내에 존재하는 전체 점포 수 (경쟁 강도).
- **`saturation` (상권 포화도)**: `local_competitors / store_age`. 신생 상권에 가게가 몰려있을수록 값이 커집니다.
- **`sales_per_store` (점포당 기대 매출)**: 지역 평균 매출을 경쟁자 수로 나눈 값.

---

## 6. 모델 평가 및 불균형 처리 (Evaluation & Imbalance Handling)
수집된 데이터는 '폐업(1)'이 '영업(0)'보다 많은 불균형 상태입니다.
- **가중치 보정 (`scale_pos_weight`)**: XGBoost 학습 시 `(영업 중 데이터 수) / (폐업 데이터 수)` 비율을 곱해주어, 모델이 다수결로 찍는 편법을 막았습니다.
- **평가 지표 (ROC-AUC)**: 0/1 분류에서 Accuracy(정확도)는 클래스 비율에 따라 왜곡될 수 있습니다. 따라서 모델이 양성/음성을 얼마나 잘 변별하는지 나타내는 **AUC Score** (0.5=찍기, 1.0=완벽)를 핵심 지표로 삼습니다. 현재 Baseline은 약 0.77 수준입니다.

---

## 7. 팀원들을 위한 확장 가이드 (How to Extend New Data)
추가 데이터(예: 지하철역 거리, 젠트리피케이션 지수 등)를 실험 모델에 붙이는 방법입니다.

1. **데이터 Fetch (data_acquisition.py)**: 신규 API 호출 함수를 작성하고 로컬 CSV/Pickle로 캐싱합니다. (매번 API를 호출하면 차단될 수 있습니다.)
2. **키(Key) 기반 조인 (feature_merger.py)**: `create_master_table` 함수 내부에서 신규 데이터를 병합합니다.
   - 공간 기반 데이터(역세권 등): `BAS_ID`를 기준으로 조인.
   - 행정구역 기반 데이터(임대료 등): `BAS_ID`의 앞 5자리(`SIG_CD`: 시군구)를 추출하여 조인.
3. **피처 투입 (main.py)**: 병합이 완료되면 `exclude_cols`에 해당 컬럼이 들어가지 않도록 주의하여 모델에 투입합니다.

---

## 8. 모델 성능 한계 돌파를 위한 튜닝 가이드 (Tuning Strategy)

### A. 하이퍼파라미터 튜닝 (`baseline_xgboost.py`)
현재 설정된 모델은 과적합 방지에 초점이 맞춰져 있습니다. 다음 값을 변경하며 CV(Cross Validation) 점수를 확인하세요.
- **`max_depth` (현재 4)**: 5~7로 늘리면 모델이 더 복잡한 패턴을 학습하지만 과적합 위험이 커집니다.
- **`learning_rate` (현재 0.05)**: 0.01로 줄이고 `n_estimators`를 500 정도로 늘리면 미세한 패턴 학습이 가능합니다.
- **`reg_lambda` / `reg_alpha`**: L2/L1 정규화 수치. 과적합이 발생할 경우 이 값을 키워 모델에 패널티를 부여하세요.

### B. 피처 엔지니어링 아이디어
- **업종별 차별화**: 현재는 전체 업종을 묶어 계산하지만, '카페'와 '음식점'의 생존 주기(`store_age`) 기준은 다릅니다. 업종(대분류)을 One-Hot Encoding하여 피처로 추가해 보세요.
- **계절성 및 시계열**: 폐업이 특정 분기(연말, 연초)에 몰리는 경향이 있는지 `close_month` 피처를 만들어 검증해 보세요. (단, 모델 학습 시점에서는 결측치 처리에 유의해야 합니다.)

### C. 치명적인 주의사항: 데이터 누출 (Data Leakage)
팀원들이 파생 변수를 만들 때 가장 흔히 하는 실수입니다.
- 모델은 **"현재 시점의 정보"**만 가지고 **"미래의 폐업"**을 맞춰야 합니다.
- 만약 피처 중에 `폐업일자`, `생존기간(survival_months)`, `폐업전매출` 같은 미래 지표가 하나라도 섞여 들어가면 모델 점수가 99%로 뛰는 '데이터 누출(Leakage)'이 발생합니다. 새로운 컬럼 추가 후 점수가 비정상적으로 높다면 누출을 의심하세요.
